IMPORTANT LEGAL AND RULES NOTE

BattleTech names, rules, and setting material belong to their respective rights
holders. This project document is an implementation plan, not a substitute for
official rulebooks. Rules must be verified against legally owned official source
material before release. Sarna and MechWiki are useful lore/index references but
are not authoritative rules sources.

Status labels used in this document:

  APPROVED       A current project decision.
  PROPOSED       A recommended design awaiting owner approval.
  VERIFY         Must be checked against the selected official rules edition.
  OPTIONAL       A rule or feature controlled by a setting.
  FUTURE         Outside the initial release scope.
  DEFERRED       Intentionally postponed.

Requirement identifiers are stable. Do not renumber existing identifiers when
adding new entries. Add new identifiers at the end of the relevant series.

1. PROJECT OVERVIEW
----------------------------------------------------------------------

1.1 Vision

Develop a comprehensive BattleTech game system for Foundry VTT that recreates
tabletop BattleTech play while using virtual-tabletop automation to reduce
repetitive work. The system should remain understandable to new players,
powerful for experienced players, and modular enough to support future campaign
and unit expansions.

1.2 Initial Product Goals

  GDD-001  Preserve player and Game Master control while automating arithmetic.
  GDD-002  Show calculations and modifiers instead of hiding them.
  GDD-003  Support configurable official, optional, and campaign house rules.
  GDD-004  Keep pilots separate from the BattleMechs they operate.
  GDD-005  Provide responsive combat information without panel overlap.
  GDD-006  Support long-term campaign play through money, repair, and refit tools.
  GDD-007  Keep features modular so later unit types can be added safely.

1.3 Primary Reference Categories

  A. Official BattleTech rulebooks selected by the project owner.
  B. Foundry VTT Version 14 documentation and API references.
  C. Sarna.net for lore, terminology, factions, and source discovery.
  D. MechWiki for secondary lore and equipment reference.
  E. Rideable module documentation for mounted-token integration.

Exact book titles, editions, errata dates, and page references must be entered
in the rules reference before rules-dependent features are marked release-ready.

2. PRODUCT VISION AND DESIGN PRINCIPLES
----------------------------------------------------------------------

2.1 Design Principles

  DP-001  Rules transparent: show the components of rolls and totals.
  DP-002  Automation reversible: authorized users can correct mistakes.
  DP-003  Source traceable: every implemented rule has a source or house-rule tag.
  DP-004  Data safe: transactions and combat changes are logged when practical.
  DP-005  UI resilient: variable equipment lists must not overlap other panels.
  DP-006  Accessible: do not communicate state by color alone.
  DP-007  Extensible: avoid hard-coding Great Houses, weapons, or technology eras.
  DP-008  Performant: recalculate derived data only when dependencies change.
  DP-009  Compatible: isolate optional module integrations behind adapters.

2.2 Automation Levels (PROPOSED)

  Manual:     The system displays information; users make all changes.
  Assisted:   The system previews changes and asks for confirmation.
  Automatic:  The system applies valid changes and records an audit entry.

Each major subsystem should declare which levels it supports.

3. SCOPE, USERS, AND PLATFORMS
----------------------------------------------------------------------

3.1 Initial Users

  Game Master: full configuration, combat, economy, and correction authority.
  Assistant GM: delegated campaign and combat management.
  Trusted player: expanded token and sheet controls as permitted.
  Unit owner: controls owned pilots and BattleMechs.
  Observer: read-only access to permitted information.

3.2 Initial Release Scope

  - Foundry VTT Version 14 system foundation.
  - BattleMech and Pilot actors.
  - Responsive HUD and sheets.
  - Ammunition, heat, weapon groups, and GATOR assistance.
  - Construction-weight validation for explicitly supported equipment.
  - Rideable integration when the module is installed.
  - M-Notes, mission rewards, and a basic Mech Bay/storefront.

3.3 Explicitly Future Scope

Vehicles, infantry, battle armor, ProtoMechs, aerospace units, VTOLs,
DropShips, WarShips, strategic maps, manufacturing, and complete career mode.

4. USER INTERFACE AND HUD
----------------------------------------------------------------------

4.1 HUD Themes

  UI-001  Use the current HUD palette as the default for independent mercenaries,
          unaffiliated pilots, neutral units, and units without a faction.
  UI-002  Add data-driven themes for House Davion, House Kurita, House Liao,
          House Marik, and House Steiner.
  UI-003  Automatically select a theme from the unit or company faction.
  UI-004  Allow a user or GM override without changing the underlying faction.
  UI-005  Theme records include colors, contrast-safe text, accents, and optional
          heraldry assets whose licensing is suitable for distribution.
  UI-006  Confirm faction palettes through approved references before release.

House associations:

  House Davion  - Federated Suns
  House Kurita  - Draconis Combine
  House Liao    - Capellan Confederation
  House Marik   - Free Worlds League
  House Steiner - Lyran Commonwealth / Lyran Alliance, depending on era

4.2 Responsive HUD

  UI-010  The HUD is resizable, scrollable, and optionally dockable.
  UI-011  Piloting information must never overlap the ammunition display.
  UI-012  Panels wrap, collapse, or scroll at constrained sizes.
  UI-013  Remember window size, position, docking, and collapsed panels per user.
  UI-014  Handle different BattleMech equipment loads without fixed-height assumptions.
  UI-015  Provide a reset-layout command.
  UI-016  Support keyboard navigation and visible focus indicators.

4.3 HUD Information

  - Pilot identity and relevant skills
  - BattleMech identity, chassis, and variant
  - Armor and internal structure condition
  - Current heat and heat capacity indicators
  - Movement and initiative state
  - Ammunition by compatible bin
  - Weapon groups and Alpha Strike
  - Current GATOR step and combined modifier
  - Current target and range band
  - Pilot condition and active effects
  - Validation warnings that affect deployment or firing

4.4 GATOR Presentation

  UI-020  Show Gunnery, Attacker Movement, Target Movement, Other Modifiers,
          and Range as visible steps.
  UI-021  Show current, completed, and remaining steps.
  UI-022  Show active unit, current step modifier, and combined modifier.
  UI-023  Display the tracker in the HUD and Combat Tracker.
  UI-024  Show a configurable phase-change notification.
  UI-025  GM controls: next, previous, skip, reset, and manual override.

Note: GATOR is primarily a calculation mnemonic/workflow. Whether it is treated
as a combat sub-phase is a project UI decision and must not silently alter the
official turn sequence. See RULE-GATOR-001.

5. COMBAT SYSTEMS
----------------------------------------------------------------------

5.1 Weapon Groups

  CBT-001  Every BattleMech sheet supports WG1, WG2, WG3, and Alpha Strike.
  CBT-002  A weapon may be assigned to one or more groups.
  CBT-003  The HUD reads group membership from the BattleMech actor.
  CBT-004  Each group previews weapons, damage potential, heat, ammunition use,
           availability, and supported range information.
  CBT-005  Firing a group rolls attacks, adds heat, consumes ammunition, updates
           actor/HUD state, and writes a combat-log entry.
  CBT-006  The preview distinguishes potential damage from expected or resolved damage.
  CBT-007  Alpha Strike contains the weapons explicitly assigned to that group;
           whether it defaults to all weapons is configurable.
  CBT-008  The system rejects or warns about impossible fire before mutation.

5.2 Ammunition Display and Consumption

Display format: CURRENT / MAXIMUM, with a compatible ammunition type and bin.

Examples from the current project requirements:

  SRM-4:          20 / 20; one four-missile volley results in 16 / 20.
  AC/10:          10 / 10; one attack results in 9 / 10.
  Gauss Rifle:     8 / 8; one attack results in 7 / 8.
  Machine Gun:  1000 / 1000; current project rule consumes 200 rounds per
                machine gun fired per turn.

  CBT-010  Track ammunition by bin and ammunition type.
  CBT-011  Support multiple weapons drawing from compatible bins.
  CBT-012  Prevent attacks when the selected bins cannot satisfy required use.
  CBT-013  Preview exact bin deductions before firing.
  CBT-014  Update current ammunition without changing maximum capacity.
  CBT-015  Record weapon, quantity, bin, before value, and after value.
  CBT-016  Support empty bins, destroyed bins, CASE status, and critical effects.

The missile-count and machine-gun rules above are APPROVED project requirements
but may differ from a selected tabletop ruleset's usual ammunition accounting.
They must be implemented as clearly labeled campaign/house-rule modes unless
verified as the intended official rule interpretation. See Part IV.

5.3 Heat

  CBT-020  Preview heat for each weapon group before firing.
  CBT-021  Apply heat once for each weapon actually fired.
  CBT-022  Resolve heat effects from a versioned rules profile.
  CBT-023  Allow authorized correction with an audit note.
  CBT-024  Distinguish generated heat, dissipation capacity, and current heat.

5.4 Movement Outside Normal Restrictions

  CBT-030  Provide an OPTIONAL GM setting that allows authorized movement while
           paused, outside the unit's turn, or outside the current workflow step.
  CBT-031  Permission choices: GM, Assistant GM, Trusted, Owner, or Everyone.
  CBT-032  Record out-of-sequence movement when logging is enabled.
  CBT-033  Clearly mark this as an override; do not silently bypass Foundry controls.

6. BATTLEMECH CONSTRUCTION AND VALIDATION
----------------------------------------------------------------------

6.1 Included Mass and Configuration Inputs

  CON-001  Chassis and internal structure
  CON-002  Engine, gyro, and cockpit
  CON-003  Armor and armor technology
  CON-004  Weapons and ammunition
  CON-005  Heat sinks
  CON-006  Jump jets
  CON-007  CASE and coolant-flushing equipment
  CON-008  Electronics and miscellaneous installed components
  CON-009  Technology base, era, availability, slots, and compatibility where supported

Initial construction technologies include standard armor, Ferro-Fibrous armor,
standard internal structure, and Endo Steel. Exact names, weights, slots, and
technology restrictions must come from verified rules data.

6.2 Validation Output

  CON-010  Display maximum chassis tonnage.
  CON-011  Display current installed tonnage with defined rounding behavior.
  CON-012  Display remaining tonnage.
  CON-013  Flag overweight, incompatible, unavailable, and incomplete builds.
  CON-014  Separate warnings from deployment-blocking errors.
  CON-015  Explain each validation result and identify affected components.
  CON-016  Do not mark a build deployment-ready while blocking errors remain.

7. PILOT AND BATTLEMECH ACTOR SYSTEMS
----------------------------------------------------------------------

7.1 Separation

  ACT-001  Pilots and BattleMechs are separate Foundry Actors.
  ACT-002  Pilots retain skills, injuries, experience, biography, equipment, and history.
  ACT-003  A pilot can enter, exit, and transfer between BattleMechs.
  ACT-004  Pilot data is referenced, not duplicated into the BattleMech.
  ACT-005  The BattleMech HUD uses the mounted pilot's applicable statistics.
  ACT-006  Define behavior for no pilot, multiple candidates, deleted pilot, and broken link.

7.2 Rideable Integration

  ACT-010  Use an optional adapter for the Rideable module.
  ACT-011  When mounted, pilot and BattleMech occupy the same token location.
  ACT-012  Render the BattleMech above the pilot and hide the pilot visually.
  ACT-013  Movement of the BattleMech moves the mounted pilot.
  ACT-014  Exiting restores visibility and independent control.
  ACT-015  The core system remains usable when Rideable is absent or disabled.

8. ECONOMY, MECH BAY, AND MISSION RESOLUTION
----------------------------------------------------------------------

8.1 M-Notes

  ECO-001  M-Notes are the campaign's primary project currency.
  ECO-002  Support individual balances and a shared company balance.
  ECO-003  Successful missions pay 100 percent of the assigned reward by default.
  ECO-004  Failed missions and defeats pay 25 percent by default.
  ECO-005  The GM may set rewards, bonuses, penalties, distribution, and overrides.
  ECO-006  Every balance mutation creates a transaction record.
  ECO-007  Corrections use reversing entries rather than deleting history.
  ECO-008  Currency uses integer minor units or another exact representation,
           never binary floating-point arithmetic.

8.2 Mech Bay and Storefront

  BAY-001  View, buy, sell, repair, rearm, and refit BattleMechs.
  BAY-002  Buy weapons, ammunition, armor, heat sinks, jump jets, CASE, and equipment.
  BAY-003  Replace destroyed equipment and refill compatible ammunition.
  BAY-004  Configure weapon groups.
  BAY-005  Preview cost, tonnage, compatibility, and resulting validation before confirmation.
  BAY-006  Show chassis, variant, condition, armor, structure, heat capacity,
           equipment, repair/refit cost, completion estimate, and available M-Notes.
  BAY-007  Record purchases, sales, repairs, refits, and ammunition transactions.

8.3 Mission Resolution

  MIS-001  Outcomes: Victory, Partial Victory, Draw, Failed Mission, and Defeat.
  MIS-002  Show mission name, result, base reward, payout percentage, bonuses,
           penalties, repair costs, ammunition costs, final payout, distribution,
           and updated balances.
  MIS-003  The GM can override the default percentage and must see the final preview.
  MIS-004  Applying a resolution is idempotent or protected against duplicate payout.
  MIS-005  Reopening a completed resolution does not silently pay it again.

9. SETTINGS, PERMISSIONS, ACCESSIBILITY, AND DATA SAFETY
----------------------------------------------------------------------

  SYS-001  World settings select rules profile, era, optional rules, and automation level.
  SYS-002  Client settings store visual theme, layout, and notification preferences.
  SYS-003  Server-authoritative/GM-authoritative changes validate permissions.
  SYS-004  Never trust client-submitted costs, balances, or combat mutations without validation.
  SYS-005  Color themes meet practical contrast requirements and use icons/text as well as color.
  SYS-006  Destructive actions require confirmation and offer recovery where feasible.
  SYS-007  Export/import project data with schema-version checks.

10. TECHNICAL GOALS AND ARCHITECTURE
----------------------------------------------------------------------

10.1 Technology Baseline

  - TypeScript with strict compiler settings
  - Vite-based build pipeline
  - ESLint and Prettier
  - Foundry VTT Version 14 APIs
  - Automated tests and distributable ZIP packaging
  - GitHub issues, pull requests, and continuous integration

10.2 Architectural Style

Use layered, feature-oriented architecture:

  Presentation layer
    Actor sheets, HUD, Combat Tracker extensions, dialogs, notifications.

  Application layer
    Commands/use cases such as FireWeaponGroup, MountPilot, ResolveMission,
    PurchaseItem, ValidateMech, and AdvanceGatorStep.

  Domain layer
    Pure calculations and rules: mass, ammunition, heat, permissions,
    construction validation, transaction calculations, and rules profiles.

  Infrastructure layer
    Foundry documents, settings, sockets, hooks, migrations, compendia,
    module adapters, serialization, logging, and packaging.

Dependency rule: domain code must not import Foundry UI or document classes.
Infrastructure adapts Foundry data into domain inputs and persists domain outputs.

12. CORE DATA MODELS
----------------------------------------------------------------------

All records include schemaVersion where they may persist independently.
Identifiers should be stable UUIDs, not display names.

12.1 MechActorSystem

  identity: chassis, variant, customName, factionId, technologyBase, era
  capacity: ratedTonnage, currentMass, remainingMass
  movement: walking, running, jumping, movementState
  defense: armorByLocation, structureByLocation, armorType, structureType
  thermal: currentHeat, sinkCapacity, heatEffects
  equipmentRefs: ordered embedded-item identifiers
  weaponGroups: WG1, WG2, WG3, ALPHA membership and display order

12.2 PilotActorSystem

12.3 WeaponItemSystem

  ammoTypeKey, currentUnits, maximumUnits, unitMeaning

tabletop shot-based accounting and missile/round-count house-rule accounting.

12.5 EquipmentItemSystem

  technologyBase, eraAvailability, effects, operationalState

12.6 RulesProfile

12.7 GatorWorkflowState

  combatId, combatantId, attackId

  startedBy, updatedBy, timestamps, status

12.8 EconomyAccount and Transaction

  Account: id, ownerType, ownerRef, currencyKey, balanceMinorUnits

12.9 MissionResolution

13. SERVICES, APIS, HOOKS, AND MODULE INTERACTIONS
----------------------------------------------------------------------

13.1 Domain/Application Services

  MechValidationService
    Calculates mass and produces structured diagnostics.

  AmmoService
    Finds compatible bins, previews consumption, applies atomic deductions.

  WeaponGroupService
    Calculates group availability, heat, ammunition, ranges, and fire plan.

  AttackService
    Validates and resolves attacks through the active rules profile.

  HeatService
    Applies generation, dissipation, thresholds, and effects.

  PilotMountService
    Maintains bidirectional pilot/mech links and calls the Rideable adapter.

  GatorService
    Owns the assisted calculation workflow and manual overrides.

  EconomyService
    Applies exact, permission-checked, auditable transactions.

  MissionResolutionService
    Previews and commits a payout once.

13.2 Public API (PROPOSED)

Expose a namespaced, versioned API such as game.battlemechFoundry.api.v1:

  validateMech(mechUuid)
  getWeaponGroupPreview(mechUuid, groupId, context)
  fireWeaponGroup(mechUuid, groupId, targetUuids, options)
  mountPilot(pilotUuid, mechUuid, options)
  dismountPilot(pilotUuid, options)
  getGatorState(combatantUuid)
  advanceGator(combatantUuid, options)
  previewTransaction(request)
  applyTransaction(request)
  previewMissionResolution(missionUuid, request)
  resolveMission(missionUuid, request)

Mutating functions return structured results and error codes. They validate
permissions and current document versions before writing.

13.3 Foundry Hooks and Interactions

  init            Register settings, sheets, schemas, and API namespace.
  setup           Register optional integrations and configuration UI.
  ready           Run compatibility checks and safe pending migrations.
  updateActor     Refresh derived HUD data when relevant actor paths change.
  updateItem      Revalidate owning mech and refresh relevant previews.
  updateCombat    Refresh GATOR and combat-tracker presentation.
  canvasReady     Reconcile mounted-token presentation if needed.

Use narrow change-path checks to avoid full recalculation on unrelated updates.

13.4 Module Interaction Flow

Weapon group fire:

  HUD -> WeaponGroupService preview -> AttackService validation
      -> AmmoService allocation + HeatService calculation
      -> atomic/guarded Foundry document updates
      -> chat/combat log -> HUD refresh

Pilot mounting:

  Sheet/HUD -> PilotMountService -> permission/link validation
            -> RideableAdapter when available -> actor reference update
            -> token layer/visibility reconciliation -> HUD refresh

Mech Bay purchase:

  Store UI -> catalog and construction preview -> EconomyService validation
           -> transaction + item changes -> validation -> receipt/log

13.5 Rideable Adapter Contract

  isAvailable()
  mount(riderToken, mountToken)
  dismount(riderToken)
  synchronizePosition(riderToken, mountToken)
  ensureLayering(riderToken, mountToken)

Failure of the adapter must return a useful error and must not leave actor links
half-updated. Integration-version support belongs in a compatibility matrix.

14. PERSISTENCE, MIGRATION, SECURITY, AND COMPATIBILITY
----------------------------------------------------------------------

14.1 Persistence

Use Foundry Actors and embedded Items for core game state, world settings for
world rules, client settings for personal layout, and flags only for data that
cannot live cleanly in the system schema. Avoid duplicating authoritative data.

14.2 Migration

  - Increment schemaVersion for persistent shape changes.
  - Migrations are ordered, repeat-safe, and backed up/exportable.
  - Provide dry-run diagnostics for destructive or ambiguous migrations.
  - Never silently discard unknown equipment or transaction history.
  - Test migrations from every supported prior release.

14.3 Concurrency and Atomicity

Foundry document writes may not provide database transactions across documents.
Commands must use precondition checks, operation identifiers, and compensating
records where needed. Mission payout and purchasing require duplicate-operation
protection. Combat commands compare expected/current versions before applying.

14.4 Security and Permissions

  - Treat all client requests as untrusted.
  - Recalculate prices, ammo use, and rewards in the authoritative handler.
  - Check ownership and role for every mutation.
  - Validate UUID type and document scope.
  - Sanitize user-authored HTML and chat content.
  - Avoid secrets in settings, logs, or source bundles.

14.5 Compatibility

Maintain a tested matrix for:

  - Foundry VTT core version
  - Browser families supported by Foundry
  - Rideable module versions
  - Other declared optional integrations
  - Data schema and rules-profile versions

15. CODING, TESTING, PACKAGING, AND DOCUMENTATION STANDARDS
----------------------------------------------------------------------

15.1 Coding Standards

  CODE-001  TypeScript strict mode; avoid any unless justified in code review.
  CODE-002  Pure functions for rule arithmetic whenever practical.
  CODE-003  Explicit units in names/types: tons, heat, shots, missiles, rounds.
  CODE-004  No floating-point currency.
  CODE-005  No magic equipment values in UI code; use versioned rule data.
  CODE-006  Structured error/result types for user-facing commands.
  CODE-007  Internationalization keys for visible interface text.
  CODE-008  Comments explain intent and rule provenance, not obvious syntax.

15.2 Testing Strategy

  Unit tests:
    mass, ammunition allocation, heat, range, payout, permissions, migrations.

  Integration tests:
    actor/item updates, weapon-group firing, mounting, store purchase, mission payout.

  UI tests:
    resizing, scrolling, keyboard flow, HUD updates, notifications, duplicate clicks.

  Rules fixtures:
    verified examples with source edition/page, inputs, outputs, and rounding.

  Manual Foundry tests:
    GM/player roles, pause behavior, multi-client synchronization, module absent/present.

15.3 Definition of Done for a Feature

  - Acceptance criteria met.
  - Rule provenance or project-rule designation recorded.
  - Automated tests added at appropriate layers.
  - Permissions and multi-client behavior checked.
  - UI checked at supported sizes and zoom levels.
  - Accessibility basics checked.
  - Migration added if stored data changed.
  - Documentation and change log updated.
  - No unresolved release-blocking defects.

15.4 Packaging and CI

CI should install dependencies reproducibly, lint, type-check, test, build,
validate manifests, and create a deterministic installable ZIP. Releases include
checksums, supported Foundry versions, migration notes, known limitations, and a
human-readable change log.

20. RULES GOVERNANCE AND SOURCE HIERARCHY
----------------------------------------------------------------------

20.1 Source Priority

  1. Selected official rulebook and edition.
  2. Applicable official errata for that edition and cutoff date.
  3. Official supplemental book when the feature enables it.
  4. Explicit project interpretation approved by the project owner.
  5. Optional campaign/house rule.
  6. Sarna or MechWiki only for discovery/lore cross-checking, not final authority.

20.2 Required Source Register Fields

  Source ID
  Full title
  Edition/printing
  Publication identifier if applicable
  Errata version/date
  Rules profile(s) using it
  Notes about precedence or conflicts

20.3 Rule Record Fields

  Rule ID
  Feature
  Status: UNVERIFIED / VERIFIED / PROJECT RULE / OPTIONAL / DEFERRED
  Source ID, chapter, page, and errata reference
  Plain-language interpretation
  Inputs, outputs, units, rounding, and edge cases
  Implementation module/function
  Automated fixture IDs
  Approver and approval date

Do not copy lengthy copyrighted rulebook text. Record concise implementation
summaries, citations, examples derived for testing, and the project's interpretation.

21. RULE-TO-IMPLEMENTATION MAPPING
----------------------------------------------------------------------

RULE-TURN-001 - Turn and phase sequence
  Status: VERIFY
  Implementation: combat state machine and Combat Tracker integration.
  Validate: supported turn phases, initiative, activation, reset, and edge cases.

RULE-GATOR-001 - GATOR attack modifier calculation
  Status: VERIFY + PROJECT UI WORKFLOW
  Implementation: GatorService and HUD/Combat Tracker components.
  Mapping: Gunnery + attacker movement + target movement + other + range.
  Caution: UI steps must not be presented as official phases unless source confirms it.
  Validate: sign conventions, modifier stacking, minimum range, special effects,
            immobile/prone states, and manual overrides.

RULE-WEAPON-001 - Weapon attack resolution
  Status: VERIFY
  Implementation: AttackService.
  Validate: target number, dice procedure, impossible shots, hit resolution,
            weapon state, targets, and combat log explanation.

RULE-GROUP-001 - Weapon groups
  Status: PROJECT UI FEATURE
  Implementation: WeaponGroupService and BattleMech sheet.
  Interpretation: groups organize legal individual attacks; grouping must not
                  change the underlying rules unless an optional rule says so.

RULE-AMMO-001 - Standard ammunition accounting
  Status: VERIFY
  Implementation: RulesProfile + AmmoService.
  Validate: unit represented by each ammunition record, shots per ton, compatible
            ammunition types, multi-weapon fire, bin choice, dumping, and criticals.

RULE-AMMO-002 - Missile-count accounting
  Status: APPROVED PROJECT RULE; expose as OPTIONAL/profile mode
  Implementation: ammoAccountingMode = perProjectile.
  Example: SRM-4 20/20 becomes 16/20 after one full four-missile volley.
  Validate: partial fire, cluster results, streak/advanced systems, and whether
            consumption is based on launched missiles rather than missiles that hit.

RULE-AMMO-003 - Ballistic and Gauss one-to-one accounting
  Status: APPROVED PROJECT REQUIREMENT; VERIFY weapon-by-weapon.
  Implementation: weapon rule data defines ammoPerAttack.
  Examples: AC/10 and Gauss Rifle consume one configured ammunition unit per attack.

RULE-AMMO-004 - Machine gun 200-round consumption
  Status: APPROVED PROJECT HOUSE RULE
  Implementation: machineGunConsumptionMode = rounds200PerWeaponPerTurn.
  Example: two machine guns fired consume 400 rounds under this mode.
  Validate: repeat fire in one turn, variants, insufficient rounds, and damage effects.

RULE-HEAT-001 - Weapon heat generation
  Status: VERIFY
  Implementation: HeatService and weapon rule data.
  Validate: heat timing, multiple weapons, special equipment, and shutdown effects.

RULE-HEAT-002 - Heat dissipation and effects
  Status: VERIFY
  Implementation: HeatService at the correct turn boundary.
  Validate: sink types, environmental effects, thresholds, penalties, shutdown,
            ammunition explosion checks, and coolant-flushing equipment.

RULE-MOVE-001 - Movement modes and modifiers
  Status: VERIFY
  Implementation: movement domain and GATOR integration.
  Validate: walk/run/jump, distance brackets, terrain, facing, elevation, prone,
            immobile, minimum movement modifier, and special movement systems.

RULE-MOVE-002 - Out-of-turn/paused movement
  Status: OPTIONAL PROJECT OVERRIDE
  Implementation: permission-controlled movement override and audit log.
  Note: this is a VTT convenience, not an implied tabletop rule.

RULE-MASS-001 - Chassis tonnage and installed mass
  Status: VERIFY
  Implementation: MechValidationService.
  Validate: all component formulas, fractional values, rounding, maximum tonnage,
            and technology-dependent changes.

RULE-ARMOR-001 - Standard and Ferro-Fibrous armor
  Status: VERIFY
  Implementation: armor rule data and construction validator.
  Validate: mass/points, slots, location maximums, tech base, era, and variants.

RULE-STRUCT-001 - Standard and Endo Steel internal structure
  Status: VERIFY
  Implementation: structure rule data and construction validator.
  Validate: mass, slots, location restrictions, tech base, era, and variants.

RULE-EQUIP-001 - Equipment legality and placement
  Status: VERIFY
  Implementation: equipment rule data and validators.
  Validate: mass, critical slots, legal locations, quantities, conflicts, tech base,
            availability, CASE, jump jets, sinks, electronics, and coolant systems.

RULE-DAMAGE-001 - Armor/internal damage and critical hits
  Status: VERIFY; later combat scope
  Implementation: future DamageService.
  Validate: hit locations, transfer, critical checks, destruction, ammo explosions,
            CASE interaction, and pilot consequences.

RULE-PILOT-001 - Gunnery and Piloting skills
  Status: VERIFY
  Implementation: Pilot Actor plus Attack/Movement services.
  Validate: defaults, modifiers, wounds, consciousness, ejection, and skill checks.

RULE-ECO-001 - M-Notes and mission payouts
  Status: PROJECT CAMPAIGN RULE
  Implementation: EconomyService and MissionResolutionService.
  Defaults: success 100 percent; failure/defeat 25 percent; GM override allowed.
